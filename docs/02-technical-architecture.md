# 技术架构草案

## 推荐路线

TopoShell 建议采用混合架构：

- 原生宿主：C# / .NET 8。
- UI 层：WPF 或 WinUI 3。
- 3D/视差层：WebView2 + Three.js，或后续 DirectX/Win2D。
- 系统集成：Win32 API + Windows App SDK 能力。
- 配置：本地 JSON。

第一阶段优先使用 WPF + WebView2。原因是：

- 对透明窗口、置顶窗口、无边框窗口、鼠标穿透等桌面层行为更容易控制。
- 可以快速承载 Three.js 场景，复用 `sonic-topography` 这类项目的视觉经验。
- C# 负责系统能力，Web 前端负责复杂视觉，两边职责清楚。

## 进程结构

```text
│  ├─ 低亮度地形/网格

│  └─ 音频响应

```

## 窗口层级

建议先做 4 个窗口：

- `SceneWindow`：桌面底层视觉，默认不抢焦点，可鼠标穿透。
- `DockWindow`：贴屏幕边缘，置顶或普通层级可切换。
- `CommandWindow`：快捷键唤出，居中或贴边。
- `SettingsWindow`：普通设置窗口。

第一阶段不要强行把 SceneWindow 嵌进 WorkerW 桌面层。先用普通无边框窗口验证视觉和性能，再评估是否需要嵌入桌面。

## 系统集成能力

### MVP 必需

- 获取显示器尺寸和 DPI。
- 获取鼠标位置，用于视差。
- 全局快捷键。
- 打开应用和文件路径。
- 读取固定快捷方式列表。
- 最小化到托盘。
- 配置导入导出。

### 第二阶段

- 枚举运行中窗口。
- 激活窗口。
- 读取应用图标。
- 读取系统音量。
- WASAPI loopback 获取系统音频。
- 自动隐藏或弱化任务栏。

### 高风险能力

以下能力暂缓：

- Explorer 注入。
- 任务栏 hook。
- 替换 Windows Shell。
- 修改注册表 Shell。
- 驱动级输入或显示能力。

## 视觉实现

### Scene Layer

用 Three.js 实现：

- 低多边形地形或棋盘网格。
- 鼠标驱动的相机偏移。
- 低亮度扫描线。
- 慢速漂移的点阵。
- 音频驱动的局部隆起。
- 多层深度：近景 UI、中景网格、远景地形。

### UI Layer

用原生 XAML 或 WebView 前端都可以。建议：

- Dock 和 Command 用原生 WPF/XAML，保证系统集成稳定。
- Scene 用 WebView2/Three.js，保证视觉效率。
- 设置页早期可用 XAML，后续再考虑统一设计系统。

## 音频方案

第一阶段可只做内部模拟音频数据：

- 用随机平滑波形模拟低频、中频、高频。
- 验证地形响应、发光、节奏感。

第二阶段再接入 WASAPI loopback：

- 捕获系统输出音频。
- 计算 FFT。
- 拆分为 8 个频段。
- 每个频段驱动不同视觉参数。

建议频段映射：

- SUB BASS：中心抬升。
- BASS：底部厚度。
- LOW MID：大面积缓慢波动。
- MID：方向性流动。
- HIGH MID：边缘亮点。
- PRESENCE：局部闪光。
- BRILLIANCE：细线闪烁。
- AIR：轻微颗粒漂移。

## 性能目标

- 空闲 GPU 占用低于 5%-8%。
- 默认帧率 30 FPS，可切换 60 FPS。
- 笔记本电池模式自动降低动效。
- 场景可完全关闭，只保留 Dock 和启动器。
- 所有窗口都必须可关闭、可恢复。

## 安全原则

- 任何系统修改都必须可逆。
- 第一次启用任务栏相关功能前弹出明确说明。
- 默认不隐藏任务栏。
- 默认不随系统启动。
- 默认不读取浏览器、账号或隐私数据。
- 配置只写入项目定义的数据目录。
